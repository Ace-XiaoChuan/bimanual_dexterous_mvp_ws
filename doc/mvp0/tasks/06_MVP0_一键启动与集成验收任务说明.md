# MVP-0 第六阶段工作说明：一键启动与集成验收

## 1. 工作目标

在第五阶段完成 `assembly_task_node` 后，本阶段的目标是将 MVP-0 从“多个节点可以手动分别启动”提升为：

```text
一条命令可启动
一组测试可验收
连续运行可复现
失败场景可观察
```

本阶段完成后，MVP-0 应形成一个完整的纯软件闭环：

```text
Fake Scene Manager
Fake Arm Control
Assembly Task Node
StartTask 请求
TaskState 状态发布
集成测试验收
```

也就是说，本阶段不是继续增加机械臂动作复杂度，而是先把当前最小链路“关严实”。

## 2. 本阶段范围

实现：

```text
assembly_bringup 中的 MVP-0 launch 文件
assembly_tests 中的 MVP-0 集成验收测试
README 或任务文档中的一键运行说明
```

建议新增或修改：

```text
assembly_bringup/
├── launch/
│   └── mvp0_fake_system.launch.py
├── CMakeLists.txt
└── package.xml

assembly_tests/
├── test/
│   └── test_mvp0_task_flow.py
├── CMakeLists.txt
└── package.xml
```

本阶段验证的节点：

```text
fake_scene_manager_node
fake_arm_control_node
assembly_task_node
```

本阶段验证的接口：

```text
/assembly/reset_scene
/assembly/move_arm
/assembly/start_task
/assembly/task_state
```

暂不实现：

```text
完整 Pick and Place
灵巧手动作
末端操作
双臂协同
真实 MoveIt 规划
MuJoCo 物理仿真
真实机械臂驱动
任务取消
任务暂停和恢复
自动失败恢复
```

## 3. 一键启动要求

在 `assembly_bringup` 中提供一个 launch 文件：

```text
mvp0_fake_system.launch.py
```

该 launch 文件应启动：

```text
fake_scene_manager/fake_scene_manager_node
fake_arm_control/fake_arm_control_node
assembly_task/assembly_task_node
```

启动命令建议为：

```bash
ros2 launch assembly_bringup mvp0_fake_system.launch.py
```

启动后应能看到以下接口存在：

```bash
ros2 service list
ros2 action list
ros2 topic list
```

预期至少包含：

```text
/assembly/reset_scene
/assembly/start_task
/assembly/move_arm
/assembly/task_state
```

## 4. 集成测试目标

集成测试的目标不是测试某一个函数，而是验证 MVP-0 的最小纵向链路确实可用。

测试应覆盖以下场景：

```text
合法任务请求
空 task_name
不支持的 task_name
任务状态发布
连续多次执行
```

推荐测试流程：

```text
启动 MVP-0 fake 系统
等待 /assembly/start_task 服务可用
等待 /assembly/reset_scene 服务可用
等待 /assembly/move_arm action 可用
订阅 /assembly/task_state
发送 StartTask 合法请求
检查响应 accepted = true
检查最终收到 SUCCESS 状态
发送非法请求并检查错误码
重复执行多次合法任务
确认节点没有崩溃或卡死
```

## 5. 验收用例

### 用例 1：构建通过

命令：

```bash
cd ~/bimanual_dexterous_mvp_ws
source /opt/ros/humble/setup.bash
colcon build --symlink-install
source install/setup.bash
```

预期结果：

```text
所有 MVP-0 package 构建成功
没有 Python import 错误
没有接口生成错误
```

### 用例 2：一键启动

命令：

```bash
ros2 launch assembly_bringup mvp0_fake_system.launch.py
```

预期结果：

```text
fake_scene_manager_node 启动成功
fake_arm_control_node 启动成功
assembly_task_node 启动成功
```

### 用例 3：接口存在

检查服务：

```bash
ros2 service list
```

预期存在：

```text
/assembly/start_task
/assembly/reset_scene
```

检查 Action：

```bash
ros2 action list
```

预期存在：

```text
/assembly/move_arm
```

检查 Topic：

```bash
ros2 topic list
```

预期存在：

```text
/assembly/task_state
```

### 用例 4：合法任务成功

请求：

```bash
ros2 service call \
  /assembly/start_task \
  assembly_interfaces/srv/StartTask \
  "{task_name: 'mvp0_home'}"
```

预期响应：

```text
accepted = true
task_id 非空
error_code = 0
message = "Task accepted"
```

预期状态序列：

```text
RESETTING
ARM_HOME
SUCCESS
```

最终状态：

```text
current_state = "SUCCESS"
progress = 1.0
error_code = 0
```

### 用例 5：空任务名失败

请求：

```bash
ros2 service call \
  /assembly/start_task \
  assembly_interfaces/srv/StartTask \
  "{task_name: ''}"
```

预期响应：

```text
accepted = false
task_id = ""
error_code = 3001
message = "task_name must not be empty"
```

### 用例 6：不支持的任务名失败

请求：

```bash
ros2 service call \
  /assembly/start_task \
  assembly_interfaces/srv/StartTask \
  "{task_name: 'unknown_task'}"
```

预期响应：

```text
accepted = false
task_id = ""
error_code = 3002
message = "unsupported task_name"
```

### 用例 7：连续执行

连续调用合法任务 10 次：

```text
mvp0_home
```

预期结果：

```text
每次请求都返回 accepted = true
每次 task_id 递增
每次最终进入 SUCCESS
节点不崩溃
节点不死锁
日志中能看到每次状态转换
```

## 6. 日志要求

本阶段应重点检查日志是否能支撑调试。

每次合法任务至少应能从日志中看到：

```text
start_task_request
start_task_response
task_state RESETTING
reset_scene_request
reset_scene_response
task_state ARM_HOME
move_arm_goal_received
move_arm_feedback
move_arm_result
task_state SUCCESS
```

失败场景至少应记录：

```text
task_id
previous_state
current_state
error_code
message
```

日志不要求做成完整实验管理系统，但需要能回答三个问题：

```text
任务什么时候开始？
失败发生在哪个状态？
失败原因是什么？
```

## 7. 完成标准

本阶段完成后，应满足：

```text
colcon build 通过
launch 文件可一键启动 MVP-0 系统
/assembly/start_task 可被调用
/assembly/task_state 可观察状态变化
合法任务最终进入 SUCCESS
非法任务返回明确错误码
连续 10 次合法任务不崩溃
集成测试或测试脚本可以复现上述验收流程
README 中有对应运行命令
```

如果上述条件没有满足，不建议进入 MVP-1。

## 8. 与下一阶段的关系

第六阶段完成后，MVP-0 才算真正闭环。

下一阶段建议进入：

```text
MVP-1：RViz / MoveIt 单臂 Pick-and-Place
```

MVP-1 的重点应是把 `fake_arm_control` 的内部逻辑逐步替换为真实规划或仿真规划接口，但上层任务接口尽量保持稳定。

建议替换顺序：

```text
Fake MoveArm
→ MoveIt named target
→ MoveIt pose target
→ 简单 Pick-and-Place 轨迹
```

不要在 MVP-1 一开始就同时引入：

```text
双臂
灵巧手全关节控制
MuJoCo 接触
真实物体抓取
Peg-in-Hole 插入
```

这些内容应在基础链路稳定后逐步加入。

## 9. 真机过渡条件

完成第六阶段后，仍不建议直接上真机执行完整任务。

可以开始真机准备工作的最低条件：

```text
MVP-0 一键启动和集成验收通过
MVP-1 中 RViz / MoveIt 单臂回 home 流程稳定
真实机械臂驱动接口已经单独验证
急停、限速、关节限位和工作空间边界明确
任务可以安全停止或取消
空载动作已经低速验证
```

可以开始真机 Pick-and-Place 的建议条件：

```text
单臂仿真 Pick-and-Place 稳定
真实机械臂低速空载动作稳定
抓取区域、放置区域和障碍物边界明确
末端执行器开合动作单独验证
任务失败时机械臂不会继续执行危险动作
现场有人监控并能够立即急停
```

真机过渡建议顺序：

```text
真机单节点 bringup
→ 单臂低速回 home
→ 单臂低速移动到固定安全点
→ 空载执行简化任务
→ 软物体或安全道具 Pick-and-Place
→ 再考虑更接近装配的任务
```

不要从当前 MVP-0 直接跳到：

```text
双臂真机协同
真实物体抓取
Peg-in-Hole 插入
接触力控制
```

当前阶段的正确目标是先证明软件链路可靠，而不是证明真实装配能力。
